# 架构深度审查报告 (Architectural Audit) - Deployer

## 1. 核心架构模式分析

### 1.1 单例容器模式 (Singleton Container)
- **实现**: `Deployer\Deployer` 作为全局唯一的上下文持有者。
- **优点**: 简化了配方文件中的全局函数（如 `set`, `get`, `task`）调用。
- **风险**: 
    - **测试困难**: 难以在同一进程内运行多个独立的部署上下文。
    - **状态污染**: 全局配置可能被意外修改，导致不可预知的执行结果。

### 1.2 组合式任务流 (Composite Task Flow)
- **分析**: Deployer 8.x 使用基于 Collection 的任务管理。任务不再是简单的数组，而是具有拓扑顺序的执行链（before/after 钩子）。
- **优化建议**: 文档应明确任务的“生命周期图谱”，包括预编译阶段和运行时阶段。

## 2. 分布式系统挑战

### 2.1 最终一致性与符号链接
- **问题**: 在多 Web 服务器集群中，`current` 链接的切换并非严格同步。
- **架构建议**: 引入“二阶段提交”概念，或在负载均衡层通过心跳检测确保所有节点的版本一致性。

### 2.2 状态持久化 (Host Storage)
- **发现**: Deployer 将主机状态（如 `release_name`）存储在 `Deployer\Host\Storage`。
- **架构审计**: 对于长连接或断点续传，缺乏外部持久化支持（目前主要在内存中）。

## 3. 安全架构审查

### 3.1 凭证管理
- **现状**: 依赖 SSH Agent 或明文配置。
- **建议**: 应该在文档中增加关于 **Secrets Management** (如 HashiCorp Vault 集成) 的架构建议，而非仅仅依赖环境变量。
