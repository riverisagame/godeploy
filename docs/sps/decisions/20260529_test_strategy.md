# 测试补齐架构审计与策略 (2026-05-29)

## 1. 核心需求对齐
用户要求：**对现有所有的功能补齐测试，包括后端功能、前端界面，确保功能完整、紧固、能用。**

## 2. 潜在影响与架构弱点 (自我攻击 & 性能对冲)
- **后端 (Go)**:
  - `engine.go`: 并发任务调度逻辑是否真的紧固？如果两个不同项目同时构建，有没有竞态条件？如果强制中断 (kill)，孤儿进程是否会彻底清理？
  - `git.go`: `EnsureRepoCache` 的并发克隆是否做了锁机制？
  - `db.go`: 测试执行过程中的数据隔离？如果现在采用 SQLite 本地文件，是否有 `.db` 泄漏或多并发锁死 (`database is locked`) 的风险？
- **前端 (Vue + Vitest)**:
  - `Dashboard.vue`: WebSocket 断线重连机制是否被测试覆盖？如果后端长时间不响应，前端状态机的降级情况如何？
  - ElTree 的全选/半选状态在初始化回显时的确切行为。

## 3. 测试隔离方案 (物理零污染)
- 后端：所有涉及 DB 的操作必须使用 `mode=memory` 并且在 `t.Cleanup` 阶段释放，禁止任何 `DROP/TRUNCATE` 持久化表。所有 Git 测试环境必须采用临时构建的 Git bundle/mock server，禁止访问真实的 Github/Gitlab。
- 前端：使用 `vi.mock` 和 `msw` 彻底阻断任何对 `localhost:8080` 的真实网络请求。

## 4. 行动准则
进入测试驱动开发模式前，必须分模块进行规划：
1. **模块A (后端调度核)**：锁与并发测试，模拟高频请求与任务超时。
2. **模块B (后端Git与网络)**：异常流处理，超时挂起。
3. **模块C (前端状态机)**：WebSocket 中断、恢复、乱序消息的鲁棒性。
4. **模块D (前端UI树与表单)**：新加的排除文件与备注提交。
