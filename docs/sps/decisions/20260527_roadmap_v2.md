# 架构决策与路线图对齐 (Roadmap V2)

## 1. 需求分析与对抗性拷问 (Adversarial Critique)

用户提出同时实现四大维度的巨量需求：业务扩展(Webhooks/RBAC/多节点)、性能调优、前端优化、底层重构。

**架构师判定：**
**拒绝以大杂烩方式同时开工。** 一口气实现四大维度的需求是典型的瀑布流灾难，会直接引发以下恶性副作用：
1. **代码合并冲突**：底层重构与上层业务（RBAC）强耦合，同时修改会导致测试基线全面崩塌。
2. **上下文污染**：UI 优化与高并发压测属于完全不同的上下文，多线程开发容易造成状态机紊乱。
3. **违反受控原则**：这严重违反了纳米级计划（10-20行变更）与绝对 TDD 的约束。

## 2. 拟定的受控执行序列 (Sequential Milestones)

我们必须以“小步快跑、严格验证”的模式，将大需求拆解为四个独立的 Milestone（里程碑），逐个击破：

### Milestone 1: 基础设施强化与底座重构 (对应需求 4 & 2)
**目标**：为高并发和多节点打好底座。
- **动作 1**：将单体 `SSHExecutor` 重构为可扩展的接口池，支持并行 SSH 连接。
- **动作 2**：编写 Benchmark 压测（大文件并发分发、大量空任务并发排队）。
- **影响**：可能破坏现有的 `engine.go` 执行路径，需极其严苛的 TDD 验证。

### Milestone 2: 核心业务扩展 (对应需求 1)
**目标**：接入外部系统与细粒度权限。
- **动作 1**：实现 RBAC 权限（Admin / Deployer / Viewer），修改 `Auth` 中间件与 DB 结构。
- **动作 2**：Webhooks 触发器，暴露 `/api/webhooks/github` 接口，并自动排队任务。
- **影响**：涉及 SQLite DML/DDL 变更，需要编写数据迁移脚本（Migration）并确保无损。

### Milestone 3: 多节点集群部署 (对应需求 1 扩展)
**目标**：一次任务下发至多台物理机。
- **动作**：修改 `Project` 表支持多 Host 绑定，改写 `RunDeploy` 实现并发分发与最终一致性状态机（全部成功才算 Success）。
- **影响**：极容易引发并发竞态，且回滚（Rollback）逻辑复杂度呈指数级上升。

### Milestone 4: 前端交互深度优化 (对应需求 3)
**目标**：全异步、动态化的高端 UI。
- **动作 1**：将现有的短轮询任务状态替换为 WebSocket 实时推送，减少 HTTP 开销。
- **动作 2**：增加 Framer Motion 或 Anime.js 等动效，提升部署流水线的视觉反馈。
- **影响**：前后端接口范式变更，需重写前端网络层。
