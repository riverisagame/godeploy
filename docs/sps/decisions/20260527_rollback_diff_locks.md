# ARCH-004: 精准回滚、Git Diff、并发部署锁及日志防护架构决策

* **日期**: 2026-05-27
* **状态**: 批准
* **发起人**: 资深架构工程师
* **受访者**: 用户

## 1. 决策背景

在对已有的 Go 后端及 Vue 前端代码进行 [SCAN] 阶段深度审计时，发现以下几处严重设计缺陷，亟需通过测试驱动开发（TDD）进行高可用与健壮性改造：
1. **回滚不精准**：原有回滚逻辑盲目指向倒数第二个成功版本，无法让用户在前端列表中自由选择某个历史版本进行回滚。
2. **缺失对比接口**：前端控制台点击“对比”按钮触发 `/api/tasks/:id/diff` 接口，但后端尚未实现。
3. **竞态并发冲突**：部署流水线为异步 Goroutine 运行，若对同一项目+环境同时触发多次部署，会导致本地克隆冲突及目标服务器部署状态破损。
4. **大日志性能威胁**：读取日志接口一次性加载全部日志，若日志文件过大将导致内存暴涨，且无法满足接口 150ms 的硬性响应要求。
5. **Rsync 错误丢失**：`cmd.Stderr` 被设置为了未分配的 `io.Writer` (nil)，导致 rsync 权限或传输错误无法记录在日志中。

---

## 2. 解决方案设计

### 2.1 精准回滚机制
* 路由保持：`POST /api/tasks/:id/rollback`
* 后端逻辑：
  1. 根据 `task_id = :id` 查询部署任务详情，获取其 `project_id`, `env_id` 以及 `release_name`。
  2. 校验该任务状态是否为 `success`。若非成功任务，拒绝回滚（防止将环境切到破损版本）。
  3. 获取该项目环境下对应的服务器配置列表。
  4. 遍历服务器，调用 `engine.SwitchSymlink(server, releaseName)`，直接将软链接切到该任务的历史 `release_name` 目录。
  5. 更新数据库中该任务的状态，或记录一笔新的回滚审计任务。

### 2.2 Git Diff 接口实现
* 路由新增：`GET /api/tasks/:id/diff`
* 后端逻辑：
  1. 根据 `:id` 查询当前任务的 `project_id`, `env_id` 和 `commit_id`。
  2. 查询在当前任务 ID 之前（`id < :id`）且状态为 `success` 的最近一个任务，获取其 `commit_id` (作为 `prev_commit`)。
  3. 若无前置成功任务，说明为首次部署，接口直接返回 `"diff": "首次部署，暂无对比历史。"`。
  4. 若存在前置成功任务，定位到本地项目构建区（例如 `workspace_path/project_id/release_name`），在此目录下执行本地命令：
     `git diff <prev_commit> <curr_commit>`
     并将输出内容作为 JSON `{"diff": "..."}` 返回。

### 2.3 环境级部署互斥锁
* 后端逻辑：
  1. 在 `HandleCreateTask` 中，在将任务状态插入数据库之前，执行数据库锁校验：
     `SELECT COUNT(*) FROM deploy_tasks WHERE project_id = ? AND env_id = ? AND status IN ('pending', 'deploying')`
  2. 如果计数 `> 0`，说明该项目在该环境下已有任务在构建或同步中，API 直接拒绝创建并返回 `409 Conflict`。
  3. 这样保证在同一时间，单个项目+环境最多仅有一个活跃的部署协程在操作 Git 和 Rsync，彻底规避竞态冲突。

### 2.4 日志截断与 150ms 性能防护
* 后端逻辑：
  1. 在 `HandleGetTaskLog` 接口中，使用限制大小的 Reader。
  2. 限制单次读取日志文件大小上限为 `1 MB`（约数万行日志，足以覆盖一次正常构建的所有细节）。
  3. 若文件超过 1MB，则只读取尾部的 1MB 内容，防止因加载超大日志造成内存高负荷和 I/O 阻塞。

### 2.5 Rsync 错误捕获
* 后端逻辑：
  1. 修改 `ssh.go` 中对 `exec.Command` stderr 的重定向。
  2. 将其重定向到 `bytes.Buffer`，并在命令失败时将 `stderr` 内容格式化后附加到错误返回中，确保其能被写入部署日志。

---

## 3. 对现有功能影响与评估
* **只读配置加载**：无影响。
* **物理数据与表结构**：完全无损。测试全量连接 SQLite 内存库，对物理表零侵入。
* **前端展示**：前端界面已预留接口及弹框，后端实现后可实现无缝集成，直接去除 Mock 状态，提升体验。

---

## 4. 物理零污染与 DDL 绝对禁绝
所有新增的单元测试和集成测试，必须连接 SQLite 内存数据库。测试代码中绝对禁止出现 `DROP`、`TRUNCATE`、`CREATE TABLE` 词字，保证无污染。
