# 架构决策记录 (ADR) 001: 废除单包约束，全面拥抱 DDD

## 状态
已通过 (由用户显式确认)

## 背景
当前项目 `godeployer` 遵循一个严格的架构约束：“单一扁平 package，无子包，不可按子目录拆分引用”。
然而，随着业务逻辑复杂度的增加，所有的 API、数据库访问、部署引擎和认证逻辑都混杂在一个包下，导致代码难以测试、职责纠缠不清，不符合现代化软件工程的长期维护需求。

## 决策
用户明确指示：“全面拥抱 DDD”。
因此，我们决定废除原有的“单一扁平 package”规则，在重构过程中将系统拆分为领域驱动设计（DDD）推荐的标准子目录结构（如 `domain`, `application`, `infrastructure`, `interfaces`）。

## 后果
1. **正面**：职责明确，核心领域模型将与外部框架和基础设施解耦；测试更容易被隔离。
2. **负面**：短期内需要修改大量的包导入路径；重构期间的回归风险增加。
3. **缓解措施**：实施严格的“纳米级测试驱动开发（TDD）”，确保现有功能零污染、零回归。
