# 需求深挖与对齐 (SCAN): 极致测试覆盖率与全配置项级文档化

## 1. 需求解析 (Requirements Parsing)
**原始诉求**：
1. 测试尽可能百分百覆盖。
2. 文档尽可能把每个配置都写的无比详细，让大家一看就懂。

**需求细化与对齐**：
- **测试覆盖率目标**：
  - **后端 (Go)**：目前处于 ~62.8% 覆盖率。需要针对 `api.go`, `db.go`, `ssh.go`, `notifier.go` 中尚未覆盖的边界异常、并发条件、鉴权拒绝等逻辑增加大量表驱动测试 (Table-Driven Tests)。目标 > 90% 并逼近 100%。
  - **前端 (Vue)**：目前处于 ~58.33% 覆盖率。需要针对 Vue 组件 `Login.vue`, `Dashboard.vue`, `History.vue` 增加基于 `vitest` + `@vue/test-utils` 的单元/组件测试，覆盖所有的界面交互逻辑（如按钮防抖、空状态、请求失败重试）。
- **文档化目标**：
  - **配置文件 (`config.yaml`)**：需要一份专门的文档（或在 `README.md` 中），逐字段解释：项目名称、描述、环境变量字典、集群节点（Host, Port, User, SSH Key, 密码）、部署脚本（本地构建、远程前置/后置）、回滚策略（保留版本数）、Webhook 订阅事件、定时任务配置等。不仅要有含义，还要给出示例和注意事项（例如密码与密钥互斥，优先使用密钥等）。

## 2. 影响范围评估与对冲审计 (Adversarial Audit & Impact Radius)
**自我攻击与性能对冲**：
1. **测试冗余与执行时间膨胀**：
   - *攻击*：如果强行追加几百个 100% 覆盖率测试用例，是否会拉垮 CI 时间？
   - *对冲*：对于 Go，需引入 `t.Parallel()`；对于 Frontend，确保 `vitest` 并发执行。测试必须保证物理隔离（Mock DB/Mock SSH），严禁操作真实远端服务器。
2. **测试维护性地狱 (Fragile Tests)**：
   - *攻击*：过度耦合内部实现细节的单元测试会导致重构困难。
   - *对冲*：Go 测试只测公共接口和 API HTTP 响应，前端测试只测 DOM 用户行为与渲染结果（Behavior Driven），而不是直接断言内部状态数据。
3. **文档与代码脱节**：
   - *攻击*：文档写得再详细，若后期增加新配置项，极易过时。
   - *对冲*：在 Go `Config` 结构体以及前端的 `types` 定义中增加高可读性的结构体 Tag 或行内注释。

## 3. 待确认列表 (Open Questions)
1. 对于“尽可能百分百覆盖”，考虑到网络 IO 与进程调用的极特殊异常（如进程中途被操作系统强制 OOM Kill），我们是否设定一个务实且极高的目标（例如 > 90% 核心逻辑分支），而不是为了硬凑 100% 写无效断言？
2. 文档是整体补充到现有的 `README.md` 中，还是单独拆分出一份 `CONFIGURATION.md` 并在 `README.md` 中引用？（建议单独拆分以防 README 过于臃肿）。
