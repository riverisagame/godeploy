# Deployer 开发文档 (Advanced Development Guide)

## 1. 核心架构哲学
Deployer 的架构设计遵循 **“配置即代码，任务即拓扑”** 的原则。它通过一个轻量级的依赖注入容器（单例实现）管理整个部署上下文。

### 1.1 执行生命周期 (Execution Lifecycle)
Deployer 的执行分为三个严格隔离的阶段：
1. **定义阶段 (Definition)**: 加载 `deploy.php` 和 Recipes，将 `task()` 和 `host()` 调用注册到 `Collection`。此时不产生任何 IO 动作。
2. **编译阶段 (Compilation)**: 
   - **拓扑排序**: 递归解析任务的 `before()` 和 `after()` 钩子，生成有向无环图 (DAG)。
   - **主机绑定**: 将任务与目标主机集进行映射。
3. **执行阶段 (Execution)**: 
   - **并行调度**: 根据 `-p` 参数选择 `ParallelExecutor` 或 `SeriesExecutor`。
   - **原子切换**: 执行 `deploy:publish` 任务完成软链切换。

## 3. 深度组件解析
### 2.1 连接器抽象层 (Connector Interface)
Deployer 通过 `Deployer\Executor\ExecutorInterface` 抽象了执行逻辑。
- **SSH 抽象**: 依赖 `symfony/process` 封装本地 SSH 指令。
- **配置覆盖**: 主机配置 (`set`) 在运行时通过 `Host\Storage` 进行层级覆盖（全局 -> 环境 -> 主机）。

### 2.2 状态管理与单例风险
`Deployer\Deployer` 单例模式在架构上是一把双刃剑：
- **风险**: 任务闭包中访问全局 `get()` 可能获取到已被其他并发任务污染的状态。
- **架构约束**: 核心贡献者必须确保所有 stateful 操作都通过特定主机的 `Storage` 隔离。

## 3. 分布式系统考量
### 3.1 集群一致性 (Cluster Consistency)
在多节点环境中，Deployer 默认不提供强一致性保障。
- **问题**: 不同主机完成 `deploy:publish` 的时间点存在微小差异。
- **架构建议**: 在大规模环境下，建议在部署脚本末尾加入“负载均衡预热”或“外部一致性锁”机制，确保流量切换时所有节点版本对齐。

## 4. 扩展开发与设计模式
### 4.1 任务组合模式 (Composite Tasks)
利用 `task('group', ['task1', 'task2'])` 实现逻辑解耦。
### 4.2 拦截器模式 (Hooks)
通过 `before()` 和 `after()` 实现非侵入式的流程扩展。
